I have a social publishing platform with Next.js 15 (App Router, TypeScript, Tailwind CSS v4, shadcn/ui) on the frontend, FastAPI (Python 3.12+) on the backend, and Supabase (PostgreSQL 17) as the database. The Supabase Project ID is cbilnvuvvjnwyinfuhat.

The project scaffolding is complete with page stubs, sidebar, and design system. I need to implement the full authentication flow using Supabase Auth where the frontend talks directly to Supabase for auth and the backend validates JWT tokens for API requests.

The architecture pattern is: Frontend sends signIn/signUp to Supabase Auth, which returns a JWT to the Frontend. The Frontend then sends the Bearer token to FastAPI Backend, which validates the JWT and executes business logic against the Supabase DB. Passwords never touch FastAPI — Supabase handles all credential management.


DATABASE SETUP (Supabase):

Create a profiles table that extends auth.users with these columns:
- id as UUID primary key referencing auth.users(id) with cascade delete
- username as unique text, not null
- display_name as text, optional
- bio as text, optional
- avatar_url as text, optional
- role as text defaulting to 'user', with a check constraint allowing only these 7 values: user, verified_creator, community_moderator, senior_moderator, admin, compliance_officer, analyst
- account_status as text defaulting to 'active', with a check constraint allowing: active, suspended, banned, deactivated
- is_verified as boolean defaulting to false
- mfa_enabled as boolean defaulting to false
- created_at as timestamptz defaulting to NOW()
- updated_at as timestamptz defaulting to NOW()

Create a database trigger that automatically creates a profile entry when a new user signs up via Supabase Auth. The trigger should generate a username from the email prefix (for example, "john" from "john@example.com"), handle username collisions by appending a random suffix, and set display_name from user metadata if available.

Create a permissions table with columns: id as auto-generated UUID primary key, role as text, resource as text, action as text, with a unique constraint on the combination of role, resource, and action.

Seed the permissions table with a comprehensive permission matrix covering all 7 roles.

Enable Row Level Security (RLS) on both tables with appropriate policies.

GET /api/auth/me: Validates the Supabase JWT token and returns the user's profile from the profiles table. The response model should be UserProfile with id, username, display_name, bio, avatar_url, role, account_status, is_verified, created_at, and updated_at.

PUT /api/auth/me: Allows users to update their own display_name, bio, and avatar_url. Users cannot change their own role or account_status through this endpoint.


ACCEPTANCE CRITERIA:

A user can sign up with email and password and a profile is auto-created in the database. A user can log in with email and password and is redirected to the home feed. Google OAuth sign-in works by redirecting to Google and then back to the app. The JWT token is sent as a Bearer token on all API requests. The useAuth() hook returns the correct user, session, and profile. Unauthenticated users are redirected to /login. A loading state is shown during auth initialization with no UI flash. Error messages are displayed for invalid credentials.
